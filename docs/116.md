## Summary

PulsusDB has no WAL — durability doc states "All durable state lives in ClickHouse; PulsusDB processes are stateless" (`docs/architecture.md:20`). The only net for a failed ClickHouse insert is a local disk spool. Three gaps combine to make data loss silent and effectively permanent under realistic outage conditions.

## Background

Write path: `crates/pulsus-write/src/writer/{buffer,table,spool,config,error}.rs`, shared core for `LogWriter`/`MetricWriter`/`TraceWriter`. Insert client: `crates/pulsus-clickhouse/src/client.rs`. HTTP ingest handlers: `crates/pulsus-write/src/ingest/http.rs`.

- Sync mode (default, no `X-Pulsus-Async` header): handler `.await`s flush; ClickHouse failure -> `500`; upstream OTel/remote-write client retries. This path is correct.
- Retry budget on ClickHouse insert failure is small: `RETRY_MAX_ATTEMPTS = 5` (`writer/config.rs:21`), exponential backoff+jitter, base 100ms/cap 10s — a few seconds total, not "for the duration of the outage." After that, batch is spooled and dropped from the live buffer (never requeued, to avoid duplicate inserts on uncertain commits).

## Issue 1 — Async mode never reports flush failure (highest severity)

`crates/pulsus-write/src/ingest/http.rs:116-121` (and equivalent metrics/traces/remote-write handlers): when `X-Pulsus-Async: 1` is set, `sink.admit()` returns `202 Accepted` the instant the batch is appended to the in-memory buffer. No waiter is registered on the eventual flush outcome.

If ClickHouse is down when that generation's background flush task runs later, the batch silently spools to disk (see Issue 2/3) with **zero signal back to the producer**, which already received `202` and has no reason to retry.

**Impact**: any deployment/producer defaulting to async mode loses data on any ClickHouse blip, with no observable failure at the point of ingest.

**Suggested fix**: reject async-mode admission with a fail-fast/retryable response when ClickHouse is in a confirmed-unhealthy state (tie into existing `/ready` health check, `crates/pulsus-server/src/ops.rs`), rather than accepting into the buffer and silently spooling later.

## Issue 2 — Spool is audit-only, never automatically replayed

`crates/pulsus-write/src/writer/spool.rs:98`: "AUDIT-ONLY. This directory's contents are NEVER automatically replayed." Reiterated in `writer/rows.rs:428` and `:909`. Confirmed via repo-wide grep — no code path anywhere reads spooled `poison/`/`uncertain/` JSON files back into ClickHouse.

**Impact**: even sync-mode's spooled batches (after the ~few-second retry budget in Issue above is exhausted) require a human to manually inspect and replay. No reconciliation loop exists. In practice this means spooled data is "logged as lost" operationally unless a runbook + engineer intervene per incident.

**Suggested fix**: at minimum, a manual-trigger CLI/admin-API to replay a spool file with proper idempotency/dedup handling (spool entries include enough context to detect already-applied vs not, given the uncertain-insert dedup concern that caused this design in the first place — see `docs/schemas.md §2.2/§8`).

## Issue 3 — Spool volume defaults to ephemeral `emptyDir`

`deploy/charts/pulsusdb/values.yaml:151-161`, `deploy/charts/pulsusdb/templates/_helpers.tpl:461-489`: the Helm chart mounts the spool directory as `emptyDir` by default; `pulsusdb.spool.persistence.enabled` (PVC) is opt-in and off by default.

**Impact**: pod reschedule/OOM/eviction/rolling-deploy during or shortly after a ClickHouse outage — plausible, since the same infra pressure that took ClickHouse down often correlates with pod churn — wipes the one audit trail (Issue 2) that would let ops recover the data. Combined with Issue 1, this is the most concrete silent-and-permanent data-loss path in the system.

**Suggested fix**: flip Helm chart default to PVC-backed spool, or fail loudly (refuse to start / alert) if `emptyDir` is used and the spool directory is non-empty on shutdown/restart.

## Secondary concern (not a data-loss bug, noted for context)

Reader and writer share a single `ChPool` (`crates/pulsus-server/src/serve.rs:296-345`, default `pool_size: 8`). During a ClickHouse slowdown, writer flush/retry attempts can hang near the 120s insert timeout (`client.rs:117-122`) and starve the shared semaphore, degrading unrelated read queries (LogQL/PromQL/TraceQL/label-cache refresh) at the same time. Consider a separate pool or reserved permits for the write path.

## Acceptance criteria

- [ ] Async-mode ingest surfaces ClickHouse-unhealthy state to the producer instead of unconditionally returning `202` (fail-fast or retryable error when health check indicates ClickHouse is down)
- [ ] A documented, tested mechanism exists to replay spooled `poison`/`uncertain` batches back into ClickHouse without risking duplicate inserts
- [ ] Helm chart defaults to a persistent (PVC-backed) spool volume, or startup/shutdown explicitly warns/fails when spool is on ephemeral storage with unflushed content
- [ ] BDD coverage (per repo test standard) for: async-mode ingest during simulated ClickHouse outage: producer observes failure; spool replay round-trip; pod restart with PVC-backed spool preserves poison/uncertain batches

## Architecture opinion — durability boundary for ingest

Opinion for the owner to decide on; not an implementation plan. Any resulting work goes through normal plan-review. Verified against source (file:line below), not docs.

### Verified facts this rests on
- Stateless contract is explicit: "All durable state lives in ClickHouse. PulsusDB processes are stateless" (`docs/architecture.md:20`).
- The write path is **health-blind**: nothing under `crates/pulsus-write/src/` reads a CH health/ping signal. The `/ready` ping lives in `pulsus-server` (`ops.rs:54-97`) and never gates admission. Async admit returns `202` the instant the batch hits the in-memory buffer, no flush waiter (`ingest/http.rs:116-121`, mirrored in metrics/traces/remote-write).
- Pre-send retry is tiny and pre-send-only: `RETRY_MAX_ATTEMPTS=5`, base 100ms/cap 10s (`writer/config.rs:21-26`). `insert_block` downgrades every *post-send* retryable failure to the non-retryable `InsertUncertain` (`clickhouse/src/client.rs:103-141`), so during an outage a batch exhausts the ~few-second pre-send budget, spools, and is settled-and-dropped, never requeued (`writer/table.rs:363-395`).
- Spool is audit-only by design; nothing replays it (`writer/spool.rs:90-117`, `writer/error.rs:24-34`). Reason: sample tables are plain `MergeTree`, exact-once *only* by writer batch atomicity (`docs/schemas.md:31,662`); a replayed insert with unknown commit fate permanently inflates rollup-tier `val_sum`/`val_count` through materialized views — irreversible (`docs/schemas.md:173`; `client.rs:81-102`).
- Helm spool default is `emptyDir`; PVC is opt-in/off (`deploy/charts/pulsusdb/values.yaml:150-163`).
- Reader and writer share one `ChPool` (default `pool_size: 8`, `model.rs:142`); the writer's `insert_block` acquires from it with a 120s timeout (`query_timeout` default, `model.rs:79`; `client.rs:104,117`). A stalled flush can starve reads.

### 1. Is PulsusDB the wrong layer for a WAL? — Yes.
An in-process WAL is durable local state that must survive restarts and bind to a specific pod. That directly contradicts `architecture.md:20` and turns every writer into a stateful node with its own recovery lifecycle, defeating free scale/restart/load-balancing. Worse, a WAL does not solve the problem it's built for: replaying a WAL entry whose CH commit fate is unknown is the exact `InsertUncertain` hazard — the WAL just relocates the uncertain-insert / tier-inflation problem without closing it. PulsusDB should stay stateless; durability belongs to a tier that is *designed* to be durable (a queue in front, or ClickHouse behind).

**Queue-in-front (producer → durable queue → ingest-consumer → CH, commit offset only after confirmed insert) is architecturally sound** and keeps PulsusDB stateless — the queue plays the same role for the write path that ClickHouse plays for storage. For the *queued* path it does eliminate all three #116 gaps: no silent `202` (the producer's durability ack is the queue, not PulsusDB); no reliance on an ephemeral spool (the queue is the replay log on its own replicated storage); no manual-audit dead-end (redelivery = re-consume from last committed offset).

### 2. But queue-in-front is NOT sufficient on its own.
Offset-commit-after-insert is **at-least-once**, and it reproduces the uncertain-insert window exactly: consumer inserts a block, the insert times out with unknown fate, offset is *not* committed, restart re-consumes and re-inserts. It closes the loop only when paired with **ClickHouse-side idempotency**:
- Traces/log read paths already tolerate at-least-once (`span_id,kind` read-time dedup + `uniqExact`, `schemas.md:459,491,513`; raw metric read dedups `(fingerprint,timestamp)`, `schemas.md:173`).
- **Metric rollup tiers are the unrecoverable case** — materialized views cannot dedup; a redelivered batch inflates aggregates permanently (`schemas.md:173`). Closing this requires deterministic, offset-keyed block framing + `insert_deduplication_token` on Replicated tables so a re-consumed offset range produces a block CH's dedup drops *through the MV*. That is real design work, not free.
- **Post-queue fan-out** matters: one ingest unit fans out to multiple non-transactional inserts (samples + streams/series + metadata). Metadata/streams are `ReplacingMergeTree` and replay-safe by construction; the *sample* insert is the one needing the dedup token, and the offset must be committed only after it confirms.

Net: queue-in-front **+ deterministic offset-keyed insert-dedup** is the correct durable architecture. Without the dedup half you trade silent loss for silent tier inflation. Note this can be deployed with **zero PulsusDB WAL code** — a collector Kafka/Redpanda exporter→receiver tier in front of sync-mode ingest — with the CH-dedup hardening being the only PulsusDB-side work; a native offset-committing consumer is the tighter-integrated alternative if PulsusDB should own the consumer.

### 3. Direct-push correctness fixes to do REGARDLESS of the queue decision
The no-queue path (OTLP/remote-write straight in) will always exist. These are separable and independent of durability strategy:
- **Issue 1 (async silent 202) — fix.** Genuine correctness bug: async advertises "accepted" but can silently drop. Fail-fast when CH is *known* unhealthy (reuse the existing `/ready` ping signal; gate at the server sink layer, which owns both the health state and the sink — not in `http.rs`'s state-agnostic core) with a retryable `503`/`429`, AND document async as best-effort/at-most-once. The fail-fast only narrows the window (TOCTOU: a blip starting *after* admission still spools), so the doc change is the load-bearing half; **sync stays the durability-bearing default** (already returns `500` → upstream retries; the issue confirms this path is correct).
- **Issue 3 (Helm emptyDir) — fix, trivial.** Default to PVC-backed spool, or fail-loud when spool is on ephemeral storage and non-empty at shutdown. The audit spool exists even with a queue in front (uncertain inserts still land there), so PVC is the right default either way. Pure ops-safety, no hot-path risk.
- **Secondary — shared read/write `ChPool` starvation — fix (aligns with the query-perf mandate).** A stalled 120s flush hogging the 8-conn shared pool degrades unrelated reads. Give the writer a small dedicated pool or reserved permits. Real availability bug on the read path; earns its keep regardless of #116's outcome.

### 4. Decline as scope creep / wrong layer
- **In-process WAL — decline.** Contradicts the stateless contract; inherits the uncertain-insert problem without solving it; duplicates what a durable queue does better.
- **Automatic spool replay (Issue 2) — decline the automatic loop.** Auto-replaying `uncertain/` is unsafe by construction (permanent tier inflation); auto-replaying `poison/` re-attempts deterministic failures. The audit-only spool is *correct*. A **manual** admin CLI to replay a specific `poison/` file after human triage is a bounded, optional convenience (the issue's own "at minimum") — but it is not a durability mechanism, must never be automatic, and must refuse `uncertain/` unless gated behind the same insert-dedup token as (2). Low priority, off #116's critical path.

### Recommendation / option split
- **Now (direct-push correctness, any topology):** Issue 1 fail-fast + async best-effort docs; Issue 3 PVC default / fail-loud; write/read pool isolation.
- **Durable-ingest design (belongs in a queue-in-front proposal, not a WAL):** external queue tier (collector Kafka/Redpanda, or native offset-committing consumer) **paired with** deterministic offset-keyed blocks + `insert_deduplication_token` on Replicated tables so redelivery is idempotent through materialized views. This is the correct home for "no data loss during an outage."
- **Decline:** in-process WAL; automatic spool replay (manual `poison/`-only CLI is optional, low priority).
